# nyaya-memory-chat

A **standalone** legal-research chat for Indian case law that, unlike a stateless
RAG bot, **never forgets**. It uses **[Cognee](https://docs.cognee.ai)** as a
self-hosted hybrid **graph + vector memory** layer so it (a) builds a persistent
knowledge graph over our judgments corpus and (b) remembers every user across
infinite sessions.

> **Cognee retrieves · Claude composes · a grounding guard verifies.**
> The corpus knowledge graph carries our *curated* citation edges; the
> user-facing legal answer is generated by Claude `claude-opus-4-8`; and every
> citation Claude emits is checked against what was actually retrieved before it
> reaches the user.

It is a **sibling** of the existing `lex-ai/backend` monolith. It **reuses the
same Supabase Postgres** that holds the judgments (treated as a **read-only**
source of truth) and writes only into its own `memchat` schema. It never modifies
the existing backend and never writes to the judgment tables.

---

## Architecture

```
                 ┌──────────────────────────────────────────────┐
   user ──chat──▶│  FastAPI  /api/chat  (NDJSON streaming)       │
                 │                                               │
                 │  1. recall_user()  ── per-user memory ────────┼─▶ Cognee / memchat
                 │  2. retrieve()     ── corpus grounding ───────┼─▶ Cognee graph + hybrid RRF
                 │  3. Claude opus    ── compose the answer ─────┼─▶ Anthropic
                 │  4. ground_answer()── verify every citation ──┤
                 │  5. stream + BackgroundTask remember()        │
                 └──────────────────────────────────────────────┘
   READ-ONLY  public.judgments_metadata / judgment_vectors / judgment_citations
   WRITE-ONLY memchat.sessions / messages / user_facts / memory_* / cognee_ingest_log
```

Two **distinct LLM roles**, kept strictly separate:

| role | model | where |
|---|---|---|
| Cognee internal graph extraction | OpenAI `gpt-4o-mini` (`LLM_API_KEY`) | inside `cognify`/`remember` |
| user-facing legal answer | **Claude `claude-opus-4-8`** | our `app/llm.py` |
| cheap query-rewrite / fact extraction | `claude-sonnet-4-6` / OpenAI `gpt-4o-mini` | our code |
| embeddings | OpenAI `text-embedding-3-large` **@ 1536 dims** | matches `judgment_vectors` |

### Memory backends

`MEMORY_BACKEND` selects the engine:

- **`cognee`** — the real Cognee 1.0 hybrid graph+vector memory. Per-user chat
  memory via `cognee.remember()` / `cognee.recall()`; the corpus graph built with
  the **low-level Tasks → Pipeline → DataPoints** API so our curated
  `judgment_citations` edges land as real graph edges (not LLM guesses). Requires
  `pip install "cognee[postgres]"` **and** a dedicated `cognee_db` database.
- **`local`** *(default here)* — a Postgres-native fallback in the `memchat`
  schema. Facts are extracted from user turns, embedded with the **same** OpenAI
  model, and recalled by cosine similarity, strictly scoped per `user_id`. The
  corpus graph is served directly from `public.judgment_citations`. This keeps
  the service **fully runnable and testable** without provisioning a separate
  Cognee database.
- **`auto`** — try `cognee`, fall back to `local`.

> **Why `local` is the default.** Supabase's managed Postgres does not let an
> ordinary role `CREATE DATABASE`, so a dedicated `cognee_db` (required so Cognee
> never collides with the judgment tables) cannot be provisioned on the shared
> instance from here. The Cognee code path is complete and selected by
> `MEMORY_BACKEND=cognee` once a `cognee_db` (or a separate Postgres / Neo4j) is
> available. Either way the corpus **citation graph is real** — it comes from
> `judgment_citations`, the differentiator the brief asks for.

## Setup

Requires **Python 3.12** (3.11–3.13 fine; 3.14 lacks some prebuilt wheels).

```bash
cd nyaya-memory-chat
python -m venv .venv && . .venv/Scripts/activate        # Windows
# or:  python3.12 -m venv .venv && source .venv/bin/activate
pip install -r requirements.txt

# Optional — the real Cognee engine:
pip install "cognee[postgres]"      # then set MEMORY_BACKEND=cognee + a cognee_db
```

Configuration is read from this folder's `.env` **or the shared `lex-ai/.env`
one level up** (see `.env.example`). The app auto-fixes the pasted
`DATABASE_URL="DATABASE_URL=..."` double-prefix typo and normalises the scheme.

### Run

```bash
uvicorn app.main:app --reload --port 8000
# open http://localhost:8000  -> the chat UI
# http://localhost:8000/docs  -> OpenAPI
# http://localhost:8000/healthz
```

On startup the app connects the asyncpg pool, applies `migrations/001_memchat.sql`
(creating the `memchat` schema), asserts embeddings are 1536-dim and that
Cognee's target DB is **not** the judgment DB, then picks the memory backend.

### Ingest the corpus graph

```bash
python -m scripts.ingest_corpus --limit 200          # newest-first
python -m scripts.ingest_corpus --since 2015-01-01 --limit 500
```

Idempotent (skips ids already in `memchat.cognee_ingest_log`), batched
(~20 judgments / LLM call), and runs **offline** — never in the request path.
With `MEMORY_BACKEND=cognee` this runs the low-level pipeline; with `local` the
corpus graph already lives in Postgres, so the job validates + records the log.

## API

| method | path | purpose |
|---|---|---|
| `POST` | `/api/chat` | `{user_id, session_id?, message}` → **streaming NDJSON** (see below) |
| `GET`  | `/api/sessions/{user_id}` | session history |
| `GET`  | `/api/sessions/{user_id}/{session_id}` | messages in a session |
| `GET`  | `/api/memory/{user_id}` | structured "what I remember about you" |
| `GET`  | `/api/graph/{user_id}` | the user's memory subgraph (nodes/edges) |
| `GET`  | `/api/graph-corpus/{judgment_id}` | citation subgraph from `judgment_citations` |
| `GET`  | `/healthz` | status, db availability, active memory backend |

**`/api/chat` stream events** (one JSON object per line):

```jsonc
{"type":"meta",    "session_id":"…", "recalled":["…facts…"]}
{"type":"sources", "sources":[{judgment…}]}        // populates the right rail
{"type":"token",   "text":"…"}                      // streamed answer tokens
{"type":"final",   "answer":"…", "sources":[ids], "warnings":[…], "memory":{…}}
{"type":"done"}
```

`warnings[]` carries the grounding verdict: `unverified_citation` (hallucinated
precedent) and `overruled` (relied-on precedent no longer good law).

## The grounding guard (`app/grounding.py`)

The single most important trust feature. It regex-extracts Indian citation shapes
from the model's answer — `AIR 1978 SC 597`, `(2023) 1 SCC 1`, `2024 INSC 1`,
`2019:DHC:1234`, and `X v. Y` case names — and fuzzy-matches each against the
`case_title` / `citation` of the **retrieved** judgments. Anything not present is
flagged `unverified`. Relied-on judgments whose
`current_law_status.ratio_still_good_law` is `false` are flagged `overruled`.

## Tests

```bash
pytest -q
```

- `tests/test_grounding.py` — hallucinated-citation detection, overruled-law
  warning, reporter-citation normalisation (pure Python, no network).
- `tests/test_recall.py` — cross-session memory survives a brand-new session,
  per-user **isolation**, and **bounded context** (recall is capped at *k* facts
  regardless of history length). Runs the real `memchat` SQL with a deterministic
  fake LLM; auto-skips if Postgres is unreachable.

## Notes / gotchas discovered against the live database

- **505 judgments**, 6 566 pages, 40 602 vectors, 2 131 citation rows. Only **2**
  citations have a non-null `cited_id` — the rest reference *external* cases via
  `cited_citation` + `context`, so the corpus graph models cited authorities as
  their own nodes. Only **2** judgments are currently overruled.
- The deployed `public.hybrid_search` RPC has a bug: with an empty `statutes`
  filter it excludes **every** row (`array_length('{}',1)` is `NULL`, not `0`).
  We do not modify the read-only function; `JudgmentRepo.direct_hybrid_search`
  performs an equivalent read-only vector+BM25+RRF query against
  `judgment_vectors` and is used as the reliable retriever (the RPC is still
  attempted first for parity).
- Supabase **direct** connections are deprecated; use the **session pooler** host
  (`statement_cache_size=0` for pgbouncer). See `.env.example`.

Pinned: `cognee` is intentionally **not** in `requirements.txt` (heavy /
version-sensitive); install it explicitly and record `cognee.__version__` here
when you enable `MEMORY_BACKEND=cognee`.
